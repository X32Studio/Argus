# Argus Frontend ↔ Storage Contract

This is the **canonical spec** of:
- which paths the frontend fetches
- where on disk the engine writes content
- exactly what schema each file must follow

Every component (loop methodology, `/argus init`, sync script, dashboard) MUST follow this. Schema drift here is the #1 cause of "the dashboard shows nothing even though I have data."

If you're an LLM modifying any Argus file: **read this contract first and conform to it.** If you find a real reason to change the contract, edit THIS file FIRST, then propagate to the consumers.

---

## 1. Frontend fetch URLs (read-only)

The React+Vite dashboard fetches the following over HTTP (dev or static). All paths are relative to the topic root.

| URL | Purpose | Format |
|---|---|---|
| `/topics/_index.json` | List of all topics (for picker at `/`) | JSON, see §3.1 |
| `/topics/<slug>/topic.yaml` | Per-topic schema + scope + concept layers + execution routes | YAML, see §3.2 |
| `/topics/<slug>/proposal.md` | Per-topic prose framing | Markdown (free-form) |
| `/topics/<slug>/indexes/knowledge_graph.json` | Nodes + edges for the dashboard graph + sidebar + inspector | JSON, see §3.3 |
| `/topics/<slug>/records/works_json/<work-slug>.json` | Detail for one work (inspector panel) | JSON, see §3.4 |
| `/topics/<slug>/records/works_md/<work-slug>.md` | Human-readable mirror of one work | Markdown (free-form) |
| `/topics/<slug>/report/main.md` | Synthesis brief, "Report" tab | Markdown |
| `/topics/<slug>/report/reference_index.md` | Reading guide | Markdown |
| `/topics/<slug>/report/iteration_log.md` | Per-iteration change log | Markdown |

The dev-mode Vite middleware (`app/vite.config.ts`) serves these directly from `topics/<slug>/...` on disk. The build-mode sync script (`app/scripts/syncResearchData.mjs`) mirrors them into `app/public/topics/<slug>/...`.

**Excluded from frontend access** (engine-internal, never fetched):
- `topics/<slug>/logs/*`
- `topics/<slug>/sources/*`
- `topics/<slug>/summaries/*`
- `topics/<slug>/indexes/master_index.jsonl` (engine indexing only — frontend uses knowledge_graph.json for navigation)
- `topics/<slug>/indexes/route_index.json` (engine state, not frontend nav data)

## 2. Storage layout (per topic)

```
topics/<slug>/
├── topic.yaml                       # canonical schema for the topic (frontend reads)
├── proposal.md                      # prose framing (frontend reads, loop reads)
├── records/
│   ├── works_json/<work-slug>.json  # one per work, frontend fetches on inspector
│   └── works_md/<work-slug>.md      # human mirror, frontend fetches on inspector
├── indexes/
│   ├── knowledge_graph.json         # ★ frontend's primary data source for graph + sidebar + headers
│   ├── master_index.jsonl           # engine-only; one line per work
│   └── route_index.json             # engine-only; per-route state
├── logs/                            # engine-only
│   ├── search_log.jsonl
│   ├── research_state.md
│   ├── cycle.txt                    # single int, the orchestrator's iteration counter (single writer)
│   └── orchestrator.jsonl           # in-session orchestrator log (per Amendment J)
├── sources/                         # engine-only (gitignored — local PDFs / snapshots)
├── summaries/                       # engine-only intermediate scratch
└── report/
    ├── main.md                      # frontend reads
    ├── reference_index.md           # frontend reads
    └── iteration_log.md             # frontend reads
```

**Single-writer rules** (per Amendment H/I/J):
- `cycle.txt`: **only** the orchestrator (skill-as-orchestrator OR `.claude/loops/<slug>.md` stub, whichever path is active) writes this. Iteration runner subagents are forbidden.
- `records/works_json/<slug>.json` and `records/works_md/<slug>.md`: written by paper-reader subagents (per-work isolation, see `.claude/loop.md` "Parallel Deep-Read" section). One subagent → one work file pair.
- `knowledge_graph.json`, `master_index.jsonl`, `route_index.json`, `search_log.jsonl`, `research_state.md`: parent only (in iteration's post-collation phase).

## 3. Schemas

### 3.1. `_index.json` (frontend fetches at `/topics/_index.json`)

Generated by `app/scripts/syncResearchData.mjs` (build mode) or by the Vite middleware (dev mode). Lists every topic under `topics/*` whose `topic.yaml` exists.

```json
{
  "topics": [
    {
      "slug": "<kebab-case-slug>",
      "name": "<Human title from meta.name>",
      "description": "<from meta.description>",
      "status": "<draft|accepted|paused|archived>",
      "created_at": "<YYYY-MM-DD>",
      "accepted_at": "<YYYY-MM-DD or empty>"
    }
  ]
}
```

### 3.2. `topic.yaml`

```yaml
meta:
  slug: <kebab-case-slug>                    # MUST match the directory name under topics/
  name: <Human title>
  description: <one or two sentences>
  status: draft|accepted|paused|archived
  created_at: <YYYY-MM-DD>
  accepted_at: <YYYY-MM-DD>                  # only when status flips to accepted

concept_layers:                              # 3-5 entries; ontology to reason in
  - key: <kebab-case>                        # ★ REQUIRED — stable identifier
    name: <Human label>                      # ★ REQUIRED
    description: <one sentence>

execution_routes:                            # 5-10 entries; search buckets / memory partitions
  - key: <kebab-case>                        # ★ REQUIRED — used as `route:<key>` in graph node id
    name: <Human label>                      # ★ REQUIRED
    description: <one sentence>
    anchor_examples: [optional list]

scope_in:   [list of strings]
scope_out:  [list of strings]
recency_floor: <year int>

record_fields:                               # 10-30 entries; per-work record schema
  - key: <snake_case>                        # ★ REQUIRED
    required: true|false                     # ★ REQUIRED
    description: <one sentence>

graph_edge_types:                            # 5-15 entries; allowed edge `rel` values
  - key: <snake_case>                        # ★ REQUIRED — used as edges[].rel value
    description: <one sentence>

report_sections:                             # 6-10 entries
  - <kebab-case-or-snake_case-key>           # plain string keys

iteration_mix:
  new_min: 1
  deepen_min: 1
  challenge_min: 1
  synthesis_every_n_cycles: 7
```

**Hard rules:**
- Every list item under `concept_layers` / `execution_routes` / `graph_edge_types` / `record_fields` **MUST** carry a `key` (machine identifier). Free-form `name`/`description` alone is NOT enough — the frontend and the loop reference items by key.
- `meta.slug` MUST equal the directory name (`topics/<slug>/`). No mismatch.

### 3.3. `indexes/knowledge_graph.json` (★ the frontend's primary feed)

```json
{
  "nodes": [
    {"id": "route:<route-key>",            "kind": "route",             "label": "<route name>"},
    {"id": "work:<work-slug>",             "kind": "work",              "label": "<title>", "year": <int>, "analysis_depth": "deep|medium|shallow"},
    {"id": "technique:<tech-slug>",        "kind": "technique",         "label": "<name>"},
    {"id": "transferable_idea:<idea-slug>","kind": "transferable_idea", "label": "<name>"},
    {"id": "pitfall:<pitfall-slug>",       "kind": "pitfall",           "label": "<name>"}
  ],
  "edges": [
    {"src": "<node-id>", "dst": "<node-id>", "rel": "<one of topic.yaml.graph_edge_types[].key>"}
  ],
  "updated_at": "<ISO timestamp>"
}
```

**Hard rules (the frontend depends on these literally):**

1. **Every node `id` MUST be `<kind>:<slug>`** — concretely `route:`, `work:`, `technique:`, `transferable_idea:`, or `pitfall:` prefix. Bare ids like `langchain` break the frontend's `routeWorkIdsFor` filter (`edge.dst.startsWith("work:")`) and silently produce empty sidebars.

2. **Every node MUST carry a `kind` field** — one of the five literals above. Use `kind`, **not `type`**. The frontend filters by `kind`. (Legacy `type` field may coexist for human-readable annotation but `kind` is the authoritative field.)

3. **`kind: work` node's slug MUST match `records/works_json/<slug>.json` filename stem.** The frontend's inspector panel fetches `/topics/<slug>/records/works_json/<work-slug>.json` when the user clicks a work node — the work-slug here is extracted from `work:<work-slug>` by stripping the prefix. If the file doesn't exist at that path, inspector shows "not found."

4. **Every `kind: route` node** corresponds 1:1 with `topic.yaml.execution_routes[]` entry. Its id is `route:<execution_routes[].key>`. The frontend's sidebar derives route order from the graph's node order — so seed routes in the same order as the yaml.

5. **Every edge's `rel` MUST be one of `topic.yaml.graph_edge_types[].key`.** Inventing new edge relations breaks the legend and route filtering.

6. **Every edge's `src` and `dst` MUST reference a node `id` that exists in `nodes[]`.** Dangling edges are silently dropped by the frontend but are a sign of corrupt data.

7. **`belongs_to_route`** is a canonical edge relation that connects `work:<slug>` ↔ `route:<route-key>`. The frontend uses this to populate the "works in this route" list in the ActiveRoute panel. **If you record a work but don't add a `belongs_to_route` edge, the work won't appear under any route in the sidebar.** Routes that have no `belongs_to_route` edges show "0 works · 0 techniques linked" — visually identical to "no research done yet."

### 3.4. `records/works_json/<work-slug>.json`

```json
{
  "<each topic.yaml.record_fields[].key>": <value>
}
```

The keys are determined entirely by `topic.yaml.record_fields[]`. Every entry marked `required: true` MUST be present. Optional entries may be omitted when unknown.

**Hard rule:** the file's **stem** (filename without `.json`) is the work's slug — it MUST match the graph node id (`work:<slug>` → `<slug>.json`). Otherwise the inspector can't fetch the right detail.

## 4. Common failure modes and their causes

| Symptom in dashboard | Likely cause |
|---|---|
| "0 routes" in header but `execution_routes` defined in yaml | No `kind: route` nodes seeded in graph. Add them from yaml at `/argus init` or on first iteration. |
| Routes appear in sidebar but each shows "0 works · 0 techniques linked" | Missing `belongs_to_route` edges. Researcher loop must add this edge when it adds a work. |
| Inspector says "not found" on a work click | Work node id doesn't match a `records/works_json/<id>.json` file. The work was added to graph but no record file was written, OR the slug differs. |
| Header counts edges but graph SVG shows nodes with no edges | Edges reference node ids that don't exist in `nodes[]`. Dangling edges. |
| New edge type doesn't render with the expected legend color | `rel` value not in `topic.yaml.graph_edge_types[].key`. Add the key to yaml OR rename the edge. |
| Topic shows under "Drafts (not yet accepted)" forever | `meta.status` never flipped to `accepted`. `/argus init` should auto-accept; manual escape: `/argus accept <slug>`. |

## 5. How to check compliance

Use the bundled validator script (`.claude/skills/argus/scripts/validate-contract.sh`). It enforces every hard rule in §3 and reports violations by category, plus offers an `--fix` mode for deterministic auto-repair.

```bash
# Validate every topic under $PWD/topics/:
bash .claude/skills/argus/scripts/validate-contract.sh

# Validate one topic:
bash .claude/skills/argus/scripts/validate-contract.sh <slug>

# Validate AND auto-fix what's deterministically fixable:
bash .claude/skills/argus/scripts/validate-contract.sh --fix [<slug>]

# Run from outside the repo (point at a different root):
bash .../validate-contract.sh --root /path/to/repo --fix <slug>
```

**Exit code:** `0` if all checked topics are compliant (or become compliant after `--fix`); `1` if any violation remains.

**What `--fix` repairs automatically (deterministic):**
- yaml: injects missing `key:` in items of `concept_layers` / `execution_routes` / `graph_edge_types` / `record_fields` (kebab for layer/route, snake for edge_type/field)
- graph: adds missing `kind` field on nodes (infers from id prefix; defaults to `technique` if id is bare)
- graph: renames node ids that lack the `<kind>:` prefix and rewires every edge endpoint to match
- graph: seeds `route:<key>` nodes that are declared in yaml but missing from the graph
- graph: drops dangling edges (any edge whose `src` or `dst` doesn't exist in `nodes[]`)

**What `--fix` does NOT touch (requires human or future-iteration judgment):**
- `kind: work` node whose `records/works_json/<slug>.json` file is missing — can't synthesize research from a script; the next iteration should either deep-read the slug or delete the orphan node
- Edges whose `rel` isn't in `topic.yaml.graph_edge_types[].key` — could be a typo (rename) or a legitimate new edge type (add to yaml); script can't guess

**When the researcher loop runs this:** Every iteration's parent runs `validate-contract.sh --fix ${TOPIC_SLUG}` as its final step (see `.claude/loop.md` "Self-validation pass" section). Residual violations are logged to `research_state.md` under `## contract_violations` and become candidates for the next iteration to fix.

**When to run it manually:** After bulk-edits to `topic.yaml` or `knowledge_graph.json`; after a migration; whenever the dashboard shows unexpected zeros or missing routes/works.

## 6. When to update this contract

Update **this file first** when:
- The frontend changes which fetch paths it uses
- The frontend changes which fields it reads from any payload
- Storage layout changes (new files, renamed dirs)
- Schema additions (new required fields, new kinds, new edge types)

Then propagate the change to:
- `templates/claude/commands/argus.md` (the `/argus init` template)
- `templates/claude/loop.md` and `loop-summary.md` (the iteration methodology)
- `templates/app/...` (the frontend reading the new shape)
- `templates/app/scripts/syncResearchData.mjs` (sync script if storage layout changed)

This single doc, kept up to date, is the **single source of schema truth** that prevents drift.

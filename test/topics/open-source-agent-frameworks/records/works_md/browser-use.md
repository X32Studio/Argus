# browser-use

## 1. What this work actually does

browser-use is an MIT-licensed Python library that turns a natural-language task into a sequence of real browser actions executed against a Playwright-controlled Chromium. The caller hands it a task string ("apply to these jobs on LinkedIn", "fill the cart on Instacart with this list") and an LLM (OpenAI, Anthropic, Gemini, Ollama, or their own ChatBrowserUse model), and the library runs a closed-loop controller: observe the page, ask the LLM what to do next, dispatch the action, repeat until done. The project is dual-shipped — the OSS library is the substrate, and Browser Use Cloud is a hosted product on top of it with profiles, recording, CAPTCHA solving, and an HTTPS SDK. At ~95.5k stars, 9.3k commits, and 126 releases by May 2026 it is one of the most visible browser-agent projects in the open ecosystem, and is named on the Apr 2026 Web Agent Benchmarks Leaderboard as "the strongest open-source baseline" when paired with GPT-4o.

## 2. Technical mechanism

The execution kernel is a DOM-plus-vision loop, not a free-form tool-call loop. The DOM state is built each step inside `browser_use/dom/service.py` by **fusing three CDP data sources** rather than walking the DOM in JavaScript: (a) `DOMSnapshot.captureSnapshot` with `REQUIRED_COMPUTED_STYLES` and `includePaintOrder=True` to get layout boxes, paint order, and computed style; (b) `DOM.getDocument(depth=-1, pierce=True)` to retrieve the full tree with shadow-DOM piercing; (c) `_get_ax_tree_for_all_frames()` which merges per-frame accessibility trees keyed by `backendDOMNodeId` for semantic labels. The three streams are joined into an `ax_tree_lookup: dict[int, AXNode]` and woven into an enhanced node tree via `_construct_enhanced_node()`. Visibility filtering combines CSS rules (`display=='none'`, `visibility=='hidden'`, `opacity<=0`) with a configurable `viewport_threshold` (~1000px) so far-off-screen elements are dropped, and interactability is decided by `ClickableElementDetector.is_interactive()` (tag name + ARIA role) plus a `getEventListeners` JS probe (skipped on pages with >10k elements). Iframes are traversed up to `max_iframes=100`, `max_iframe_depth=5`, with cross-origin frames lazily fetched and scroll offsets corrected via `total_frame_offset.x -= snapshot_data.scrollRects.x`. The result is then fed through `DOMTreeSerializer`, which **assigns fresh integer indices per serialization cycle** — element IDs are NOT stable across DOM re-renders, though a `previous_cached_state` argument supports differential updates.

The LLM sees that indexed element list plus, when `use_vision` is on, the *current* screenshot only. Vision token control is verified in `browser_use/agent/message_manager/service.py`: `create_state_messages()` builds `screenshots = []` and appends at most one current screenshot; `self.state.read_state_images = []` is cleared every step so prior frames are implicitly dropped — there is no "keep last N images" sliding window, just "last 1." The LLM must emit a Pydantic-typed action from a fixed vocabulary (`click_element_by_index`, `input_text`, `scroll`, `go_to_url`, `extract_content`, `switch_tab`, `done`, …), which the controller dispatches through Playwright over CDP. This action vocabulary *is* the tool protocol — no MCP, no OpenAI tools layer, no JSON-schema function-calling adapter.

Memory in the OSS library is short-term only. `AgentHistoryList` stores per-step `AgentHistory` items (model output, action results, `BrowserStateHistory`, metadata) and `_make_history_item()` appends every step. There is no pruning of the *history list* itself, but `_maybe_compact_messages()` will summarize older text under `MessageCompactionSettings.max_history_items` when sending prompts to the LLM. A `TokenCost` service tracks per-call cost and injects a `BUDGET WARNING: You have used {steps_used}/{step_info.max_steps} steps` string above 75% — but this is advisory only; no hard token-limit enforcement is in the code. Long-term persistence (cookies, files) lives in the Cloud product as Profiles and Workspaces rather than in the kernel.

Multi-agent coordination is absent and now source-verified: `class Agent(Generic[Context, AgentStructuredOutput])` takes one `task`, one `browser_session`, and has no message bus, no planner/worker split, no role graph; the README shows only single-Agent examples. Users compose multiples themselves by instantiating multiple Agents in parallel. Observability in OSS consists of per-step structured logs, screenshots stored to disk via `screenshot_service.store_screenshot()` under `agent_directory` (kept indefinitely on disk even though only the latest is included in the prompt), optional GIF recording, conversation dumps, and TokenCost-based cost reporting; the Cloud adds live preview and webhooks.

## 3. Why it matters for the topic's stated goals

The open-source agent-framework landscape splits along an execution-kernel axis: tool-call loops (LangChain, AutoGen, CrewAI, smolagents, LlamaIndex agents) versus computer/browser-control loops (browser-use, Skyvern, OpenManus, OpenAI Computer Use). browser-use is the cleanest open reference for the second branch's narrower variant: browser-only, DOM-indexed, Playwright-backed. With source verified, the design point is sharper than the docs alone reveal: rather than "walk the DOM and label it," the team chose to **fuse CDP DOMSnapshot + accessibility tree + selective JS event-listener probing**, which is what gives the LLM stable semantic labels (button vs. link vs. role=button div) without requiring vision. It demonstrates that constraining the action surface to "indexed DOM clicks + typed actions" is a meaningfully different design point from general pixel-level computer use — you trade generality (no native desktop apps) for orders-of-magnitude better reliability on the web, because the agent never has to resolve pixel coordinates or wait for OCR. For the topic, it anchors the browser/computer-use thread and provides a concrete comparison point for evaluating Skyvern, OpenManus, and the closed OpenAI Computer Use along axes of openness, embeddability, and action-vocabulary design.

## 4. What is reusable

- The **CDP-snapshot + a11y-tree fusion** is a directly transferable pattern for any browser agent: it is faster than walking the DOM in JS, gives semantic labels for free, and pierces shadow DOM in one call. Reusable in any language with a CDP client.
- The indexed-DOM action vocabulary is a transferable pattern: enumerate interactables, expose stable integer handles *within a single step*, constrain the LLM to act through those handles.
- The Pydantic-typed action registry is a clean way to keep LLM outputs validated and forward-compatible — adding a new action is one schema entry.
- The dual-channel state (textual element list + annotated screenshot) is reusable as a recipe for hybrid DOM+vision agents and is cheaper than pure vision while more robust than pure DOM.
- The **"keep only the latest screenshot in prompt, persist all to disk"** pattern is a clean separation of token budget vs. audit trail and is directly transferable.
- The library/SaaS split (OSS substrate + paid cloud on top) is a reusable distribution model for agent frameworks that want a sustainable maintainer org without going closed.
- Playwright-via-CDP as the substrate is reusable and battle-tested; the project effectively validates "Playwright is the right floor" for the niche.

## 5. What is not safely transferable (within this topic's scope)

- The single-agent assumption is baked in. Patterns from browser-use do not generalize cleanly to multi-agent frameworks (AutoGen, CrewAI) where role graphs and message buses dominate.
- **Element indices are fresh per serialization cycle and explicitly NOT stable across re-renders.** Any pattern that caches an index between LLM turns (e.g., "remember button [14] for later") is a footgun on dynamic SPAs; re-resolution must happen every step.
- The benchmark story is weak: BU Bench V1 is self-curated, the README shows a chart with no published numerical scores in text, and browser-use has not published WebArena / VisualWebArena / Mind2Web numbers itself. Third-party leaderboards cite it as a strong open-source baseline (with GPT-4o) but the raw numbers are not in the project's own materials. Do not transfer claimed accuracy as evidence of generic capability.
- The advisory-only "BUDGET WARNING" approach is not a real token-budget mechanism — copying the pattern into a system that actually needs to cap spend would silently overrun.
- The Cloud product's Profiles/Workspaces memory model is not part of the OSS substrate — citing them as "browser-use memory" misrepresents the library.
- The DOM-indexed action vocabulary does not generalize to native desktop or mobile-app use cases; OpenAI Computer Use and OpenManus operate in a strictly larger but harder envelope.

## 6. Evidence quality

High for technical mechanism (source-read iteration 3). The GitHub README and docs landing are well-maintained and current (release 0.12.9, May 26 2026). Activity signals are strong and verifiable: 9.3k commits, 126 releases, ~95.5k stars, MIT license, named maintainer org with two offices. Architecture claims about DOM construction, element indexing semantics, vision-frame rotation, and the single-agent design are now verified against `browser_use/dom/service.py`, `browser_use/agent/service.py`, and `browser_use/agent/message_manager/service.py`. Benchmark posture remains the weak link: BU Bench V1 is publicly visible but methodologically opaque versus academic benchmarks, the README does not publish numerical BU Bench scores in text, and no first-party WebArena / VisualWebArena / Mind2Web numbers exist; third-party visibility (awesomeagents.ai Apr 2026 leaderboard) confirms the project is competitive but does not substitute for first-party benchmarking discipline.

## 7. Concrete next experiments or hypotheses

- Compare action vocabulary head-to-head with Skyvern and OpenManus: do they converge on the same primitive set, or are there meaningful divergences (e.g. Skyvern's workflow steps vs browser-use's flat actions)?
- Hypothesis: a thin MCP server wrapper exposing browser-use's action registry would let any MCP-capable agent (Claude Desktop, Cursor, etc.) drive a browser without adopting the full Agent loop — worth a prototype.
- Hypothesis: because indices are fresh per serialization cycle, agents that re-plan with a stale element list will mis-click; quantify the rate on Gmail / Notion / Linear and compare to a re-resolution-every-step baseline.
- Hypothesis: the CDP-snapshot fusion (DOMSnapshot + a11y + getEventListeners) is the single biggest reason browser-use outperforms naive DOM-walking agents; ablate each source on a small benchmark to confirm.
- Track whether browser-use publishes a WebArena / VisualWebArena number in the next two minor releases — first-party academic-benchmark publication is the obvious next maturity step beyond BU Bench V1.
- Track whether browser-use adds first-class OpenTelemetry / LangSmith tracing in the next two minor releases — observability is the obvious next maturity step on the engineering side.
- Map the maintainer overlap, if any, between browser-use, Skyvern, and OpenManus contributor graphs to detect ecosystem clustering.

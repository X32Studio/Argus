# A2A (Agent-to-Agent protocol)

## 1. Identity & governance

- **Project**: Agent2Agent Protocol (A2A)
- **Canonical site**: https://a2a-protocol.org/
- **Canonical repo**: https://github.com/a2aproject/A2A (Google's original `google-a2a` org was retired into `a2aproject` after the LF donation)
- **License**: Apache-2.0
- **Spec version at capture**: **v1.0.0**, released 2026-03-12
- **Maintainer / governance**:
  - Originally announced by **Google Cloud** in **April 2025**.
  - Donated to the **Linux Foundation** in mid-2025; launched at **Open Source Summit North America, 23 June 2025** under the LF's **Agentic AI Foundation** (a separate LF home from the LF Projects, LLC umbrella that hosts MCP).
  - Governed by an **8-seat Technical Steering Committee**: **Google, Microsoft, Cisco, AWS, Salesforce, ServiceNow, SAP, IBM Research** — one voting representative each. Consensus-based; majority vote on quorum; significant discussion happens in GitHub.
  - Per-SDK MAINTAINERS.md files (a2a-python, a2a-java, a2a-go, a2a-js, a2a-dotnet, a2a-samples) list per-language maintainer groups distinct from the TSC — i.e. spec governance and SDK governance are separated.
- **Net**: same governance arc as MCP (vendor-launched → Linux Foundation → multi-org TSC), reached at roughly the same time but under a different LF entity.

## 2. What A2A actually is (and is not)

A2A is a **peer-to-peer HTTP protocol between agents**. Both sides are opaque agentic systems; neither is a "host". Three actors, per the spec's key-concepts page:

- **User** — human or automated initiator.
- **A2A Client (Client Agent)** — application, service, or another AI agent acting on behalf of the user.
- **A2A Server (Remote Agent)** — an AI agent or agentic system that exposes an HTTP endpoint implementing the A2A protocol.

What A2A does **not** do:

- It does not standardize how an agent talks to tools or data sources — that's MCP's job.
- It does not prescribe orchestration topology (supervisor, mesh, handoff) — that's the framework's job.
- It does not expose internal prompts, tool lists, or reasoning chains — agents are deliberately **opaque** to each other.

## 3. Wire surface

### Bindings

The v1.0 spec defines **three normative bindings** (any compliant agent picks one or more):

| Section | Binding | Notes |
|---|---|---|
| 9 | **JSON-RPC 2.0 over HTTPS** | Canonical; what all five reference SDKs implement first. |
| 10 | **gRPC** | Same semantic methods, different framing. |
| 11 | **HTTP+JSON/REST** | Plain REST mapping for environments that can't host JSON-RPC. |
| 12 | Custom bindings | Permitted as long as semantics are preserved. |

### Core methods (binding-independent, spec §3.1)

| Method | Purpose |
|---|---|
| **Send Message** | Initiate an agent interaction (request/response). |
| **Send Streaming Message** | Same, but returns a **Server-Sent Events** stream of TaskStatus / Artifact updates. |
| **Get Task** | Retrieve task state and artifacts by ID. |
| **List Tasks** | Discover tasks with filter + pagination. |
| **Cancel Task** | Cooperative cancellation. |
| **Subscribe to Task** | Re-attach an SSE stream to an existing task (resumption after disconnect). |
| **Push Notification ops** (set/get/list/delete) | Configure webhook-based async delivery for long-running tasks. |
| **Get Extended Agent Card** | Fetch authenticated, possibly higher-fidelity AgentCard (private skills for trusted clients). |

### Conceptual model

- **Task** — "a stateful unit of work initiated by an agent, with a unique ID and defined lifecycle." First-class; survives disconnection.
- **Message** — "a single turn of communication between a client and an agent, containing content and a role ('user' or 'agent')."
- **Part** — "the fundamental content container used within Messages and Artifacts. A Part holds one of: text content, a file reference (URL or inline bytes), or structured data."
- **Artifact** — "a tangible output generated by an agent during a task (for example, a document, image, or structured data)."

Key rule: an agent responds with either a **new Task** (long-running) **or** a **Message** (immediate).

## 4. Discovery — the AgentCard

The AgentCard is a JSON document describing an agent's identity, endpoint, capabilities, security requirements, and skills. Three discovery mechanisms:

1. **Well-known URI** (recommended, RFC 8615): `GET https://{domain}/.well-known/agent-card.json`.
2. **Curated registries** — central repositories, queryable by skill / tag / provider, with selective disclosure (different cards per requesting client).
3. **Direct configuration** — env vars / config files; spec explicitly notes this is inflexible.

### AgentCard schema (core fields)

- **Identity**: `name`, `description`, `provider`, `version` (used for ETag + caching).
- **Service**: `url` (A2A endpoint).
- **Capabilities**: `streaming: bool`, `pushNotifications: bool`.
- **Security**: `schemes` — API Key / OAuth2 / OIDC / Mutual TLS / HTTP Auth.
- **Skills**: array of `AgentSkill` objects with `id`, `name`, `description`, `inputModes`, `outputModes`, `examples`.
- **Signature** (new in v1.0): optional digital signature on the AgentCard — the **Signed Agent Cards** feature, addressing impersonation in the well-known-URL flow.

## 5. Security model

- **No trusted host**. Both sides authenticate the other.
- **Authentication**: per-request HTTP-layer auth using the schemes declared in AgentCard.security: **API Key, OAuth2, OIDC, Mutual TLS, HTTP Auth**.
- **AgentCard signing**: v1.0 added Signed Agent Cards — a discovering client can verify a card was issued by the claimed provider.
- **Credential hygiene**: spec "strongly recommends out-of-band dynamic credentials rather than embedding static secrets" in cards.
- **Registry ACLs**: registries may return different cards to different clients (selective disclosure for partner-only / private skills).
- **Opacity by design**: internal tool lists, prompts, and reasoning chains are not exchanged across A2A — only Messages, Parts, and Artifacts at the agent-output boundary.

## 6. Framework adoption — pressure-test

**Claim under test**: "A2A is broadly supported across major agent frameworks" (LF / Google marketing).

**Verdict**: Stronger than the analogous MCP claim, but with one conspicuous hole — **OpenAI's stack is not in**.

| Framework | A2A role | Primitives | Evidence |
|---|---|---|---|
| **langchain / LangGraph** | client + server | full task lifecycle, streaming | Cited as native A2A in LF launch + 2026 framework surveys |
| **crewai** | client + server | task delegation, AgentCard exposure | "CrewAI added A2A task delegation in 2026" — Best Multi-Agent Frameworks 2026 survey |
| **autogen** | client + server | native | Listed as native A2A in Stellagent / Galileo adoption tables |
| **llamaindex** | client + server | native via LlamaIndex Agents | 2026 surveys |
| **pydantic-ai** | **server only** | `fasta2a.pydantic_ai.agent_to_a2a(agent) → ASGI app` | pydantic-ai docs: "fasta2a≥0.6.1 ships a Pydantic AI bridge"; in-package A2A was removed in pydantic-ai 2.0 and re-shipped via external fasta2a |
| **openai-agents-sdk** | **no native support** | uses OpenAI-proprietary handoff instead | Issue #472 on openai-agents-python (opened 2025-04-10) still tracks A2A as not implemented |
| **swarm** (OpenAI) | **no native support** | OpenAI-internal handoff | Predates A2A; no bridge |
| smolagents | none (capture) | — | Not on adopter lists |
| metagpt | none (capture) | — | Not on adopter lists |
| browser-use | none (capture) | — | Different domain (browser automation) |
| skyvern / openmanus | none (capture) | — | Not on adopter lists |
| agno | none (capture) | — | Not on adopter lists |

**Pattern**: A2A adoption clusters in (a) LF-member stacks (Google ADK, LangGraph, CrewAI, AutoGen, LlamaIndex, Semantic Kernel) and (b) Python-first interop-focused stacks (pydantic-ai). **OpenAI Agents SDK + Swarm are the holdouts** — same pattern as ACP, **opposite** pattern from MCP (which OpenAI did adopt). For the topic graph: `protocol_supports` edges from langchain, crewai, autogen, llamaindex, pydantic-ai → A2A are warranted; openai-agents-sdk and swarm should explicitly **not** have such an edge.

## 7. A2A vs MCP — and what changed

| Axis | MCP | A2A |
|---|---|---|
| Layer | host ↔ server | agent ↔ agent |
| Primary use | plug tools / data into an agent | delegate work to a peer agent |
| Trust model | host owns trust; servers untrusted | both sides peers; mutual auth |
| Discovery | per-host config (incl. registries) | AgentCard at `/.well-known/agent-card.json` (RFC 8615) + registries |
| Transport | stdio, Streamable HTTP (JSON-RPC 2.0) | HTTPS + JSON-RPC 2.0 (canonical), gRPC, REST |
| Streaming | SSE on Streamable HTTP | SSE (`Subscribe to Task`) + webhook push notifications |
| Stateful unit | session | **Task** (first-class, re-attachable, persistent artifacts) |
| Foundation | LF Projects, LLC | LF Agentic AI Foundation |
| Stable since | 2025-06-18 (current 2025-11-25) | **v1.0.0** (2026-03-12) |
| OpenAI adoption | yes (ChatGPT + Agents SDK) | **no** (Agents SDK + Swarm use proprietary handoff) |

The protocols are **complementary**, not competitive. Most production agents in 2026 run both: MCP inward (tools / data), A2A outward (peer delegation). The interesting governance signal is that they reached neutral LF stewardship within ~9 months of each other, and the A2A TSC explicitly seated Google's competitors (Microsoft, AWS, Salesforce) — the same neutralization pattern MCP went through.

**Notable absorption**: ACP (IBM/BeeAI), the most direct competing inter-agent protocol through 2025, was reported merged into A2A under LF in 2026 — IBM Research now holds an A2A TSC seat, and there is effectively one open inter-agent protocol in the field.
